HTML (the NOUNS of our webpages)

  • The head element contains information about the webpage. Here are some elements you'll fine within:
    • The 'title' element defines the title for a webpage. This element is required and is extremely important for Search Engine Optimization (SEO).
    • The 'style' element defines the CSS styles associated with a website.
    • The 'base' element defines the base URL for a webpage.
    • The 'link' element connects an external resource to the HTML document.
    • The 'meta' element defines metadata such as the character set, description, keywords, author, and viewport.
  • The body element represents the visible content shown to the user. Some very common tags:
    • 'h1' thru 'h6' (headings, 1 being the most prominment -- there should only ever be ONE h1 on a page)
    • 'p' (paragraph)
    • 'ul' and 'ol' (unordered lists, and ordered lists) with 'li' (items of the lists) inside them
    • 'img' (images) which inside the tag requires a 'src' attribute (location of the image), and an 'alt' attribute (text describing the image)
    • 'a' (links; 'a' means 'anchor'), with a 'href' attribute (the URL/file location the browser will go to when clicked)
    • 'div' (an unspecific chunk, carved out in a block -- meaning content following this will be bumped onto another line beneath it)
    • 'span' (an inline chunk, meaning the chunk carved out will stay on one line, and following content can also be on that line

** A note on divs/semantics: tags like 'main', 'section' and 'nav' all behave like divs, but add more context to the specific function of those divs. Generally, this is good practice. It adds more clarity for other devs looking at your work **

CSS (the ADJECTIVES of our webpages)

Some basic CSS properties:

  • 'margin' indicates how much space we want around the outside of an element.
  • 'padding' indicates how much space we want around the content inside an element.
  • 'text-align' is the justification/positioning of the text (i.e. left, center, right)
  • 'font-weight' is the thickness of the text (can be measured in numbers by the hundred, or names like bold or bolder, e.g.)
  • 'font-family' is the specific font used (i.e. Arial, Trebuchet, Times New Roman, etc.) -- there are defaults if you don't specify
  • 'color' is the text color
  • 'background-color' is the color of the entire background space of the element

** Colors can be described in a few different ways: pre-set color names (i.e. blue), RGB[A] values, or hex values; VS Code has a very easy-to-use 'color picker' that's point-and-click**

Ways to select elements:

  • Element/tag name
  • Class name (preceding the class name with a '.') -- this is more specific than tags, but less specific than IDs
  • Id (preceding the id with a '#') -- this is the most specific

'*' (asterisk) is used to select EVERY element of a page. You can also select the broad-scope tags, like 'html' or 'body', though I haven't seen it done often

Git

Some common commands (NOT comprehensive/exhaustive):

  • git status: checks what branch we are currently on
  • git checkout -b [branch-name]: creates a new branch and switches to it
    • if the branch already exists, you dont need the '-b' flag there
  • git branch: lists the branches of your current repo, with a * indiciting the currently active branch
  • git add [file]: add a file as it currently is to your next commit -- this is also called 'staging' a file
    • the '-A' flag will stage all files in the directory.. so far we've used this each time we've 'staged'
  • git commit -m "[msg]": commit the staged content, adding a descriptive message about what you're committing
  • git clone [URL]: retrieve an entire repo from a hosted location thru the given URL
  • git pull origin main: pulls in the base branch into your feature branch to ensure you're up to date -- do this often!

    • ** MAKE SURE to be on the correct branch even when making changes on your local machine as it may throw off the synchronization between the two. JavaScript (the VERBS of our webpages) (editorialized by my arrogance!)

Basic JS data-types ('primitives'):

  • string (i.e. text; needs to be in quotes, unlike all other data-types)
  • number (self-explanatory)
    • worth noting that 'NaN' (Not a Number) is considered a 'number' data-type, despite the contradictory name.. lol.. I guess it helps prevent 'cast' errors?
  • boolean (true or false)
  • null (void.. nothing.. but do not confuse that for the number zero!)
  • undefined (this isn't 'nothing' per se; it's what it sounds like; it's nice to have, because it avoids 'crashing' the code in a lot of situations
  • there are a few others (BigInt, symbol, etc.) buuut above are the most common ones

Variables:

  • not much to say here -- if you've ever done algebra then you get it -- they're a word used to represent some data
  • when assigning one, you need to put 'var' prior to the variable name, then a "=", then the value
  • pretty sure we'll eventually be using 'let' (mutable) and 'const' (immutable) to define them, instead of 'var'.. but I could be wrong

** Quick note about order -- a JS file is read from top to bottom (intuitive), so the order of you code matters! You don't want to reference a variable before you've assigned it! **

Conditionals:

  • if/else statements which evaluate to a boolean
  • the concept of 'truthy' and 'falsy' values; it's easiest just to declare what most (or all?) the 'falsy' values are..
    • FALSY values: false, 0 (the number), "" (empty string), null, undefined, NaN
  • we learned about the STRICT in/equality ('===' and '!=='), but there's also less strict in/equality ('==' and '!=') -- not used as much though
  • whichever the first (in order) conditional evaluates to 'true', the remainder of the conditional is skipped

Arrays (*cough* LISTS *cough*, ahem, Python.. sorry):

  • sets of values, denoted by square brackets and separated by commas
  • they are 'indexed' in order -- just remember that the count starts at ZERO (not one)
  • the daya-types in arrays can be different for each item, they can even be other arrays/objects
  • yeah.. objects *cough* DICTIONARIES *cough*.. sorry, Python again -- we aren't there yet anyway; let's disregard this for now

Loops:

  • indicated by the 'for' keyword -- they usually pair with arrays, where you go through each item of an array
  • 3 parts:
    • initial expression (assigning a variable to a 'counter')
    • a condition (a boolean that must remain true to continue the loop; it'll end once the condition becomes false)
    • increment the counter (a shorthand for incrementing by one is '[vbl]++'); the plus one incrememnt is used often
  • I'm going to whine for a moment about 'for-of' loops which are much sleeker when you're stepping through an entire array; but we'll get to that in time

Functions:

  • when you know your code will be repeating an action (or series of actions) you can define a function so you don't have to type the logic out each time (this is an example of the concept 'modularizing')
  • a function needs to be defined (keyword 'function') and named, with parens following (variables can go into the parens if the function takes an 'argument')
  • after opening the curly brace, the logic of the function goes inside
  • functions can 'return' values (I believe they return 'undefined' if there's no return statement) -- but we aren't there yet
  • remember to actually have the function execute you need to call it (use the function name with parens)
